## <a name="q1"></a>Q1：dietarysupplement.ai 能不能做个 agent

**Claude：**

能，而且我想更进一步说：对 dietarysupplement.ai 来说，agent 大概率不是"要不要加"的功能，而是这个产品本来就该长成的形态。

理由很简单。现在的打分引擎本质是一个查询工具 —— 用户丢进来一个产品，拿到 formula / quality / safety 三个分。但用户脑子里真正的问题不是"这瓶值不值"，而是"我这种情况该吃什么、怎么搭、会不会跟我在吃的东西冲突"。回答后者需要的是一个会追问、会推理、会调证据的顾问。这正是 agent 的形态。一旦你这么看，打分引擎就从"产品"降级成 agent 手里的一个 tool —— 这是整个思路最关键的一次视角切换。

按 on-ramp 来铺，我会分三层（不是三选一，是同一个终点的递进）：

- **Layer 1｜咨询型** —— 对话外壳套在打分引擎外面。理解需求、追问 profile、把打过分的产品推给你。这层主要是 prompt + 把 scoring engine 当工具调，最快能上。
- **Layer 2｜方案型 (regimen builder)** —— 目标 + profile → 配伍方案 → 相互作用检查 → 剂量与服用时序 → 每一条推荐都用 PubMed/NIH RAG 落证据 → 过一遍 compliance 校验再输出。这层才是真正拉开和"通用大模型随便答"差距的地方。
- **Layer 3｜持续型** —— 长期追踪用户：依从性、定期复评、当某个成分有新证据出现时主动提醒调整、到点复购。这层把一次性推荐变成关系。

工具箱里我特意把 user profile 也放进去 —— 它既是输入（理解你是谁）也是记忆（下次不用重头问），这是 Layer 3 持续型的基础。

**最关键的一个设计决定**：因为是健康类，compliance 不能只当末端的内容过滤器，要做成 agent 的动作前置校验 —— 核心想推荐什么、想说什么，先过一遍守门，再吐给用户。这条是监管命门：FTC 的功效声称（不能暗示治病）、各国卫监不能做疾病声称、以及"不能 practicing medicine"的红线。你跨 6 个市场，这层还得能按地区切规则（美国 FDA/FTC、泰国 อย.、中国保健品声称口径都不一样）。这恰恰也是护城河 —— 通用大模型不敢这么答，你敢且合规，价值就在这。

起点我建议：Miss Pep 女性健康做第一个垂类，从 Layer 1 起步，把打分引擎包进对话里跑通，顺手验证 compliance 守门的口径。终点不变，只是先上这段匝道。

你想先从哪头切？是先把 agent 的"人设 + 边界"（也就是 system prompt 那层，决定它敢说什么、不敢说什么）定下来，还是先把某一个工具（比如打分引擎的调用接口）接通？

## <a name="q2"></a>Q2：做个 agent 上到 GitHub

**Claude：**

做一个真正能跑、结构干净、你直接 push 上去的 repo。SDK 的工具调用结构是稳定的，模型名我用当前的 `claude-sonnet-4-6`（可在 `.env` 改）。

做好了，一个能跑、测试全过、git 已初始化提交的完整 repo。打包好后，解压后 `.git` 历史都在，你直接加 remote 就能推。

里面是上一轮架构图的 Layer 1 落地：Agent 核心（Anthropic 工具调用循环）+ 人设/边界 system prompt + 四件工具（三维打分、PubMed 证据 RAG、交互检查、用户档案）+ compliance 出口守门。三维打分引擎是纯 Python、可单测的，**8/8 测试通过**，样例里 Miss Pep 优质铁剂打 97、廉价多维片打 52。

关于"上 GitHub"：推送需要你的 GitHub 凭证/token，这一步得你自己来（我不经手账号密码和 token）。仓库我已经 `git init` + 首次 commit 好了，你只差最后两条命令：

```bash
tar -xzf dietarysupplement-agent.tar.gz
cd dietarysupplement-agent

# 方式一：有 gh CLI（最快）
gh repo create dietarysupplement-agent --private --source=. --push

# 方式二：先在 github.com 建好空仓库，再
git remote add origin git@github.com:<你的用户名>/dietarysupplement-agent.git
git push -u origin main
```

跑起来：`pip install -r requirements.txt` → 填 `ANTHROPIC_API_KEY` → `python cli.py`。

两个可以马上接着做的方向：把 `data/products.json` 换成你真实的商品库（打分引擎立刻就能跑真数据），或者按市场把 `compliance.py` 拆成 US / 泰国 อย. / 中国 三套规则包。

## <a name="q3"></a>Q3：这个打分客观吗？

**Claude：**

好问题 —— 而且我得直说：**不客观**。它"透明、可复现"，但不"客观"。这两件事经常被混为一谈，差别恰恰是你这个产品成败的关键。

能看到几个硬伤，都不是 bug，而是藏在数字背后的价值判断：

- **权重是我定的，不是数据定的。** `0.45×配方 + 0.25×质量 + 0.30×安全` 没有任何客观依据，就是我拍的。换一组权重，分数和排名就会动。一个看起来精确的 "97 分"，本质是我替你做了"配方比安全更重要"这个主观取舍。
- **分档有断崖。** 叶酸 399mcg 算 "below efficacious"，400mcg 算 "within"，一毫克之差在那一项上跳 45 分。真实世界的剂量-效应是连续平滑的，我的规则是台阶状的，边界附近的产品评分会失真。
- **不在参考表里的成分被静默忽略。** 骨钙片的维生素 K2 根本没参与打分 —— 因为我的 `REFERENCE` 里没有它。也就是说，一个塞满新颖植物提取物的产品，formula 分只反映它"恰好被我收录"的那几个成分。这是最危险的一种偏差：你不知道分数漏掉了什么。
- **最关键的一条：参考表本身就是没出处的编辑判断。** `efficacious range` 这些区间是我手敲的 —— 有效"针对什么终点"？"对什么人群"？"基于哪一级证据"？现在全没标注。RDA/UL 还算有官方依据（NIH/IOM），但连这个都因国而异（美国、中国、泰国的 UL 不一样），而且我现在拿"成年非孕女性"的同一张表去给孕期产品打分 —— 人群都对不上。

所以诚实的结论：这套引擎做到了"同样输入→同样输出、每条规则可读"，但它不是"专家会一致认同的客观真值"。

我想顺带泼一盆有用的冷水：**对你这个产品，"客观"其实是个错误目标，甚至是法律陷阱**。这个行业里没有真正客观的打分 —— Labdoor、ConsumerLab、Examine、连食品的 Yuka，全都是编辑判断。你真正需要的不是"客观"，而是**可溯源、可解释、口径一致**。而且如果你对外宣称这是"客观科学评分"再给某个品牌打低分，你同时招来两个麻烦：FTC 盯你的"科学"声称，被打低分的品牌告你诽谤。把它定位成"我们的评测方法学"反而更稳、更可辩护。

要把它从"自洽"升级到"可辩护"，我会按这个顺序硬化：

1. **事实和权重分离。** 参考表里每个数字挂一个可引用来源（UL→NIH ODS，有效剂量→具体 RCT/Cochrane/EFSA 意见），权重单独拎出来当"编辑选择"公开声明、你自己署名负责。
2. **标注口径。** 每次打分要说清"这是按成年非孕女性、一般维持目的评的" —— 孕期、哺乳期得用不同的参考表。
3. **去断崖**，改成连续函数（剂量-效应曲线），边界附近平滑过渡。
4. **显式处理未知成分** —— 要么降低置信度、要么明说"X 成分未纳入评分"，绝不静默。
5. **校准：** 找几位营养/药学专家对一批产品打分，用他们的评分反过来调你的参数，而不是凭我一个人的直觉。

## <a name="q4"></a>Q4：如果不客观，不正确，甚至可以被操控，那这个打分的意义在哪里？

**Claude：**

你这一问，是把"客观"当成了价值的来源。但在这个品类里，价值从来不来自客观 —— 因为客观本身就不存在。

"哪个补剂最好"没有客观正确答案：它取决于这个人的目标、身体状况、在吃什么药，而背后的证据本身就在争议、在演进。任何号称给出客观答案的系统都在撒谎。Labdoor、ConsumerLab、Examine、米其林、FICO 信用分、穆迪债券评级 —— 全是编辑判断，全能被影响，全不"正确"，但全都有人用、有人信、值很多钱。

所以真正的问题不是"你的分 vs 客观真值"，而是 **"你的分 vs 用户本来会怎么做"**。

一个想补铁的女性，打开抖音/亚马逊，面前 40 个产品，每个都自称最好，推荐的网红都收了钱，评论一半是刷的。她没有任何办法分辨好坏。她的现实选项是：信营销、信网红、瞎猜，或者放弃自己读论文（她不会、也读不懂）。

跟"这一堆"比，一个口径一致、锚定证据、不收品牌钱的分数 —— 哪怕只有八成时候靠谱 —— 也是碾压性改善。**用户要的不是真理，是别让我在营销垃圾里淹死。** 标准不是"完美"，是"比她自己瞎搞强，而且站在她这边"。

具体说，这个分给用户三件事，没有一件需要它客观：

- **一是压缩。** 把 40 个选项变成"值得看的 3 个 + 为什么"，省时间。
- **二是可被反驳的解释。** 分数不是重点，rationale 才是 —— "这个低分是因为铁剂量不足、且没有第三方检测"。这教会了用户东西，也让她可以不同意你。
- **三是信任锚。** 一致 + 独立 + 把账摊开 = 信任。用户不需要你宇宙级正确，只需要你始终用一套公开的标准，而且这套标准向着她。

**关于"可以被操控"** —— 这是你三个指控里最致命的，我正面接，要分清三种：

- **第一种，厂商优化产品去拿高分。** 如果你的打分奖励的是真东西（循证剂量、可吸收形式、第三方检测），那"应试"就等于真把产品做好了。这是 feature 不是 bug，Goodhart 定律在这里反过来帮你。
- **第二种，运营方自己被收买** —— 品牌付钱换高分。这个才会让产品瞬间归零，而且这正是 2008 年搞垮债券评级的原因（评级机构拿被评对象的钱），也是亚马逊评论变废纸的原因。解药不是"算法更客观"，而是**结构性独立：你的收入不能来自被你打分的品牌**。这是商业模式决定，不是技术决定。
- **第三种你可能也在担心的：** 有人把 agent 聊瘫、越狱、诱导它给烂货背书。这恰恰是我把打分做成**确定性规则引擎、而不是让 LLM 凭感觉打分**的原因 —— 分数是对结构化数据跑死规则，你没法用花言巧语把它说动。

所以倒过来问：这个产品什么时候真的没意义？三种死法 —— 你卖排名（独立性没了）；分数不给理由、不可追问（变成又一个黑箱网红）；它并不比用户自己瞎搜强（没有信息增量）。

只要避开这三条，"不客观"根本不是问题 —— 它本来就不该客观，它该**透明、独立、一致、对不确定性诚实**。一个敢说"87 分，这是理由，这几点我们也没把握"的分数，比一个假装确定的"客观 95 分"值钱得多，也安全得多。

## <a name="q5"></a>Q5：这个 agent 和 Claude 有什么区别？

**Claude：**

技术上，这个 agent 的"大脑"就是 Claude —— 你在租它的推理和语言能力。这一层你不拥有，竞争对手也能租到同一个模型。所以你这一问很要害：如果只是给 Claude 套个壳，差别到底在哪？

差别在于 Claude **结构上做不到、或做不可靠**的那几件事：

1. **确定性、可审计的打分。** 你直接问 Claude"给这个补剂打分"，它会从训练分布里生成一个听起来合理的数字 —— 每次不一样、无法追溯、没有依据，而且能被话术说动（正是你上次担心的"可操控"）。你的 agent 的分来自规则引擎跑结构化数据：同样输入→同样输出，每条带 rationale。这是"一个自信的意见"和"一个能审计、能辩护的计算"的区别。
2. **实时、可验证的证据。** Claude 凭训练记忆答，有 cutoff，还会编出看着很真的引用。你的 agent 现查 PubMed，返回真实 PMID，用户能点进去核对。
3. **你的私有商品数据。** Claude 不知道你真实的 SKU、成分含量、认证。你的 agent 跑在一个你维护的真实产品库上。
4. **一套可署名、可追责的标准。** 两个用户问 Claude 同一个问题，口径都不一样，没人负责。你的 agent 对所有人用同一套公开方法学 —— 这就是上次说的信任锚。
5. **领域合规护栏。** Claude 有通用安全，但你的 agent 把"不做疾病声称、structure-function 措辞、按地区切规则"做成了出口闸门，是产品级风险管理。

**但我得把更深的一层挑明，因为你其实在绕着它问：**

**模型是大宗商品，壳才是产品 —— 可壳也能被复制。** 任何人都能调同一个 API、写一个打分引擎、接一个 PubMed。所以你真正的、长期的护城河不在 agent 代码里，而在它下面：

- **私有数据**：商品库、你自己的检测化验数据、沉淀的用户档案 —— 越难拿越值钱。
- **独立与信任品牌**：要用年来建，抄不走（上次说的"不卖排名"）。
- **分发**：你已有的六国网络、抖音渠道，这是别人最难复制的。
- **被专家校准、对外发布、持续迭代的方法学。**

agent（Claude + 工具）是交付机制，是入场券，可复制；**护城河在底下那几层**。

类比：Perplexity 也"只是"LLM + 搜索 + 引用 + UX，当年一堆人问"这不就是个壳吗" —— 是壳，**而壳就是产品**，但活下来的壳都在模型周围建了数据/品牌/分发的真护城河，没建的都死了。Claude 是引擎，你的 agent 是车，而生意是经销网络 + 品牌 + 那张别人没有的地图。

所以"用户为什么不直接用 Claude"：因为她不会写 prompt、不知道该问什么、无法核对答案、每次结果还不一样，而且 Claude 根本不认识你的产品。你的 agent 把这些摩擦抹掉，再加上 Claude 结构上给不了的东西。

最后提醒一句平台风险：**正因为大脑是租来的，模型方哪天往这方向走，你的"壳"就贬值** —— 而能扛住这件事的，还是底下那几层数据 + 信任 + 分发。钱该往那儿砸，不是砸在 agent 的聪明程度上。

## <a name="q6"></a>Q6：使用过程中能不能沉淀私有数据？

**Claude：**

能，而且这是你**最值钱、最抄不走**的一类护城河 —— 因为它随使用复利增长，竞争对手有钱也买不到。但要分清：不是所有沉淀下来的数据都是护城河，很多是大宗商品。

- **会沉淀但不算壁垒的：** 用户的基础档案（目标、年龄、在吃什么）。它让你能 personalize、提高单用户留存，但跨用户不构成壁垒，除非聚合成洞察。

**会沉淀且是真壁垒的三类：**

1. **显性选择** —— 用户看到打分后，实际点了谁、比了谁、拒了谁。只有运营产品才拿得到，它告诉你真实的决策驱动是什么。
2. **效果反馈（最值钱也最难）** —— 这个产品对她到底有没有用？复购了吗？四周后精力好了吗？有没有副作用？如果你能把这个回路闭上，你就攒下了连 PubMed 都没有的东西：**真实人群、真实产品、真实结果**的信号。这是最深的护城河。
3. **意图/提问数据** —— 用户问什么、卡在哪、反复担心什么。对改进 agent、发现未满足的需求、判断该上什么新品，全是金矿。

关键的心智模型是**飞轮**：使用 → 数据 → 更准的打分 → 更好的结果 → 更高的信任 → 更多使用 → 更多数据。多数产品建不起飞轮，是因为只做了单向采集，没闭上效果反馈那一环。所以你的优先级不是"什么都 log"，而是**刻意把反馈回路装上**。

**然后是我觉得对你最关键、别人最难复制的一点：**

这些数据对你不只是改进 app，它是你**供给侧的需求情报**。你本来就在做跨境 OEM 和抖音选品。平台上"哪个市场的女性在问什么、拒什么、什么品类没人好好供" —— 直接告诉你下一个该生产/采购什么。**这个推荐平台会变成你工厂的感知器官。** 纯软件竞品没有你的供应链，纯工厂竞品没有你的用户触点，这种垂直打通的数据优势几乎没人有。

但有几条硬约束，否则飞轮会变包袱：

- **冷启动**：飞轮起步前你没数据，所以早期的打分必须靠规则+证据本身就够好，先把第一批用户和信任拉来。数据护城河是后面长出来的，不是开局就有的。
- **隐私合规是生死线。** 健康数据在哪都是敏感数据（中国 PIPL、泰国 PDPA、欧盟 GDPR、美国各州）。你需要明确同意、数据最小化、强安全，以及一个清晰的价值交换。**你的护城河是信任，一次泄露就清零。** 跨六国还得分区域处理。
- **数据质量**：自报效果噪声大、有混杂（安慰剂、均值回归、只有特定人才反馈）。当信号用，别当证据用，更别让飞轮把偏差洗成"循证"。

落到现在的代码：`profile.py` 已经是这个的种子。下一步不是多存几个字段，而是把**价值交换和反馈回路**设计进去 —— 让用户因为有回报而愿意给数据，并刻意采集"看到分之后她选了什么"和"用了之后怎么样"这两件事。

## <a name="q7"></a>Q7：是不是对全球的补剂都可以打分？

**Claude：**

能 —— 但要把两件事分开：**"引擎能跑"和"打得准、敢对外打"是两回事**。架构可以复制扩展，知识不会均匀铺满全球。

引擎是数据驱动的（参考表 + 商品库 + 规则），换垂类、换市场理论上都能跑。难的是喂给它的知识，而这几块在全球范围极不均匀：

- **参考表按"国 × 成分类别"变** —— RDA/UL 各国不同，有效剂量还按人群和终点不同；一张女性健康的维矿表根本评不了运动用肌酸或中药复方。
- **商品数据的获取难度天差地别** —— 美国有 NIH 的 DSLD 这种结构化标签库能用，中国几乎没有干净的标签数据库，只能靠抓取/OCR/供应商。
- **合规按法域变** —— 连"什么算补剂"都不一样：美国 DSHEA、欧盟 EFSA 偏严、中国保健食品"蓝帽子"是另一套、澳洲 TGA 当药管最严、巴西 ANVISA；同一个产品在不同市场连措辞都得改。
- **证据库也不均** —— PubMed 对西方研究的成分强，对中药体系是系统性盲区。而中药/复方，你这套"剂量 vs 有效区间"的还原论模型对它部分失效，得另起一套子模型。

所以**别把"全球"当一个开关**。范围其实是三根独立的轴 —— **成分类别 × 市场 × 垂类** —— 你要做的是一格一格填，而不是一次点亮整张网。

挑你最强的那格站稳，再一次只延一根轴。对你来说起点很清楚：**维生素/矿物质（科学最干净、RDA/UL 现成）× 中国抖音 + 美国 Amazon（你有触点）× 女性健康这个垂类**。每延一格，引擎复用，只加一个数据模块 —— 那个市场的合规规则包 + 商品数据源 + 参考值。三根轴别同时推，否则就是一次点亮整张网，每一格都半生不熟。

最关键的纪律：**别声称你覆盖不了的范围**。在没有参考数据的格子里硬打分，你会自信地打错（就是前面那个"静默漏算"，放到全球规模），把你唯一的护城河 —— 信任 —— 烧掉。明确标出"我们覆盖哪些类别、哪些市场、置信度多少，之外就说不知道"，这种诚实本身就是信任牌。

反过来说，全球这件事**对你这种人比对纯软件公司更够得着**：你在六个国家有供应链触点，意味着你能比别人更快建起每个市场的商品数据和市场认知 —— 这正是上次说的、把推荐平台和你工厂打通的那条优势。所以终点是全球，只是按格子上匝道，不是缩小终点。

补一个具体的数据起点：美国那格相对好办，NIH 有个公开的膳食补充剂标签库（DSLD），结构化标签量很大，直接能喂打分引擎。中国那格没有这种现成库 —— 而这恰恰意味着，**谁先把中国市场的商品标签数据建起来，谁就握住了别人买不到的一块**。

## <a name="q8"></a>Q8：语言呢，能用全球的语言吗？

**Claude：**

分两层看，答案完全不同。

**对话层** —— Claude 跟用户说话的语言 —— **基本免费，全球可用**。底层模型本来就强多语言，system prompt 里也已经是"用用户的语言回答"。你那六个市场其实就覆盖四种语言：中文、英文（美/加/澳）、葡语（巴西）、泰语 —— 全在模型擅长的范围内。

但这恰恰是个陷阱：**流利 ≠ 正确 ≠ 合规**。agent 能用一口地道泰语跟用户聊，可如果泰国的商品数据、证据、合规规则你还没建，它会用地道泰语说出错的、甚至违规的话。**而且越流利越危险** —— 用户更信地道的输出。

**真正难的是知识层的语言**，它分几块：

- **商品标签是本地语言的。** 要打分就得解析标签，而"维生素C / ascorbic acid / vitamin C / ácido ascórbico"得归一到同一个实体。草本更糟 —— 同一个俗名在不同语言、地区指不同植物。这是**实体归一/本体问题，不是翻译问题**。
- **证据是有语言偏斜的。** PubMed 几乎全英文；中文临床和中药文献在知网/万方，PubMed 基本搜不到。所以查证据往往得用英文查、再用用户语言呈现；中药那块，最相关的证据本身就在中文里，研究范式还不一样。
- **合规规则是按"语言 × 法域"的。** 能用什么词是各市场用本地语言监管的。我现在的 compliance 只有英文正则；泰国 อย.、中国保健食品、巴西 ANVISA 的禁用措辞，既要换语言，规则本身也不一样 —— 翻译正则没用，**得重建**。

所以同一个结论又回来了：**对话能跑遍全球，知识不能**。

落到架构，纪律就一条：**别"用某种语言"去建产品，把核心做成语言中立的，语言只当两层薄壳**。

中间是**语言无关的核心** —— 规范化的成分 ID（挂 INCI / 拉丁学名 / 化学标识）、结构化分数、标了来源语言的证据。入口一层薄壳，把"本地语言标签 / 用户提问"解析成规范成分，这是真正的活，每个市场建一次。出口一层薄壳，把规范结果渲染成用户语言，基本免费。

这么分的好处：加一个聊天语言 ≈ 免费；加一个市场的商品语言 = 建一套入口映射；而打分引擎、证据库、方法学始终是**单一真相源**，不会按语言复制成 N 份各说各话。

## <a name="q11"></a>Q11：拍照一个产品让我打分

## <a name="q13"></a>Q13：如何推送到 GitHub？

> - 最终确认：`git remote -v` 显示 `https://github.com/liaoliang1981/dietarysupplement-scorer.git` ✅

## <a name="q14"></a>Q14：我想做成一个 Skill，在用大模型的时候调用

**Claude：**

好 —— 这个其实是**最贴合"模型是层、能力是你的"那套逻辑的形态**：把确定性打分引擎 + 参考数据 + 方法学 + 合规规则打包成一个 Skill，模型（Claude）在对话里按需把它调起来跑。打分还是跑确定性代码、不靠模型猜，而 Claude 负责对话和解释。而且 Skill 在 Claude.ai、Claude Code、API 里都能用。

规范清楚了：Skill = 一个文件夹，含 `SKILL.md`（frontmatter + 指令）+ 可选 `scripts/`（确定性代码）和 `references/`（数据/方法学）。我把打分引擎放进 `scripts/`、参考数据和权重放进 `references/`（正好实践之前说的"事实/权重和算法分离"），`SKILL.md` 教 Claude 何时调用、怎么跑、怎么守合规。

Skill 做好了，校验通过、打包成了 `supplement-scorer.skill`。

里面就是我们一路谈的那套，拆成 skill 的三层：

- `SKILL.md` 告诉 Claude 什么时候调它（用户贴成分表、描述产品、拍标签、问"这个值不值/选哪个" —— 不用说"打分"也会触发）和怎么用；
- `scripts/score.py` 是确定性打分引擎，Claude 把产品转成 JSON 后跑它拿真实分数，不靠模型猜；
- `references/` 放参考表和方法学，权重/数据跟算法分开（就是之前说的"事实与权重分离"）。合规、未收录成分如实标注、各国差异提醒，都写进了指令。

**怎么用：**

- **Claude.ai**：Settings → Capabilities → Skills，上传这个 `.skill` 文件。之后任意对话里，只要你贴个补剂配方或问"这个怎么样"，Claude 就会自动调它、跑引擎、给评分卡式的解释。
- **Claude Code**：把解压后的 `supplement-scorer/` 文件夹放进你的 skills 目录。
- **API / Agent SDK**：配合代码执行，把这个 skill 挂上。

这正好回到最初那句：**模型是租来的层，这个 skill 才是你的资产**。一份引擎 + 数据 + 方法学，Claude 在哪儿都能调起来。

## <a name="q20"></a>Q20：如何推广？

**Claude：**

推广这个产品，关键先想清楚一件事：**它是个"信任型"产品，信任买不来**。你卖的是"独立、循证、不偏品牌的打分"，这种东西用冷启动投广告转化极差 —— 没人会因为一条广告就信你的评分。能滚起来的是**内容、口碑、和"真有用 + 真中立"本身**。

**好消息是：你手里有别人没有的分发。** 大多数做评测工具的创业者从零观众起步；你有抖音/TikTok 电商渠道、会做内容、六国网络、还有供应链认知。这是你最大的不公平优势，推广策略要**全押在这上面**。

**最强的一招：让"打分"本身变成内容。** 补剂测评在抖音/小红书/TikTok 天然是好内容 —— "我把最火的 10 款女性铁剂打了分，结果第一名很意外""你那瓶多维其实是'仙女撒粉'（剂量不够）""40 美元的还不如 12 美元的"。每条视频用工具现场出分、给理由，结尾导流到 dietarysupplement.ai。这一招同时做三件事：**涨粉、建品牌、喂数据飞轮**。注意一个坑：公开给具名品牌打低分有诽谤风险 —— 全程用事实和方法学说话、只用结构功能性措辞（前面做的合规层这时候就值钱了）。

**几个零成本的分发面，顺手铺**：

- 把 Custom GPT 上架 GPT Store（有人搜 "supplement" 就能撞到）；
- Claude skill 也分享出去；
- `dietarysupplement.ai` 这个精确匹配域名做 SEO（给每个成分/产品建"X 补剂好不好"的页面，自带评分）；
- 小红书、Reddit r/Supplements、健身社区里高意图人群多。

**信任怎么冷启动**：你已经把方法学开源在 GitHub 了 —— 这本身就是强信任信号（"我们的打分是公开、可审计的"，直接碾压那些黑箱）。再找几个可信的营养师、药师、健身 KOL 用它、背书。

**但有一个矛盾你必须正面定下来**，否则全盘归零：**你自己卖补剂（Miss Pep、OEM）**。一个"独立评测"一旦被看成卖货方的营销工具，信任瞬间崩。两条路选一条 —— 要么品牌和治理上把"评测"和"卖货"明确分开；要么彻底透明："我们也做产品，用同一把尺子打自己、并公开结果"，甚至敢把自家产品打得不是第一。这是前面反复说的那条护城河的现实版，**推广之前就得想清楚站哪边**。

## <a name="q22"></a>Q22：服务器（EC2）

**Claude（节选）：**

好，服务器起来了（EC2 t3.small，us-east-1，正在运行，公网 IP **204.236.242.8**）。这台机器的用处很明确：**让确定性引擎跑在一个固定的 HTTPS 地址上**。这样 ChatGPT 的 Action 才能调它（Action 必须是 HTTPS），你的网页也能直接调它做真打分 —— 三个平台终于能拿到同一个 97。

**先在控制台/域名商做三件事**：

1. 分配 Elastic IP 并关联到这台实例。
2. 安全组入站规则放开：`22`、`80`、`443`。
3. DNS 加一条 A 记录：`api.dietarysupplement.ai` → 你的 Elastic IP。

**部署脚本**（自带装环境、拉代码、开机自启、自动 HTTPS）：

```bash
ssh -i 你的密钥.pem ec2-user@204.236.242.8

cat > deploy.sh <<'SCRIPT'
#!/usr/bin/env bash
set -euo pipefail
DOMAIN="api.dietarysupplement.ai"
REPO="https://github.com/liaoliang1981/dietarysupplement-scorer.git"
U="ec2-user"

dnf install -y python3 python3-pip git
sudo -u "$U" bash -c "cd /home/$U; [ -d dietarysupplement-scorer ] && (cd dietarysupplement-scorer && git pull) || git clone $REPO; python3 -m venv /home/$U/venv; /home/$U/venv/bin/pip install -q --upgrade pip fastapi uvicorn"

cat >/etc/systemd/system/dsscore.service <<EOF
[Unit]
After=network.target
[Service]
User=$U
WorkingDirectory=/home/$U/dietarysupplement-scorer/chatgpt
ExecStart=/home/$U/venv/bin/uvicorn server:app --host 127.0.0.1 --port 8000
Restart=always
[Install]
WantedBy=multi-user.target
EOF

curl -fsSL "https://caddyserver.com/api/download?os=linux&arch=amd64" -o /usr/bin/caddy
chmod +x /usr/bin/caddy
mkdir -p /etc/caddy
printf '%s {\n  reverse_proxy 127.0.0.1:8000\n}\n' "$DOMAIN" >/etc/caddy/Caddyfile
cat >/etc/systemd/system/caddy.service <<EOF
[Unit]
After=network.target
[Service]
ExecStart=/usr/bin/caddy run --config /etc/caddy/Caddyfile
Restart=always
[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload
systemctl enable --now dsscore caddy
SCRIPT

sudo bash deploy.sh
```

**验证**：

```bash
curl https://api.dietarysupplement.ai/health
curl -X POST https://api.dietarysupplement.ai/score -H 'Content-Type: application/json' \
  -d '{"certifications":["GMP","third_party_tested"],"ingredients":[{"nutrient":"iron","form":"bisglycinate","amount":18},{"nutrient":"folate","form":"5-mthf","amount":400}]}'
```

返回里 `"overall"` 应该跟 ChatGPT/skill/web 一致。

**接到 ChatGPT（让它变确定性）**：把仓库里 `chatgpt/openapi.yaml` 的 `servers.url` 改成 `https://api.dietarysupplement.ai`，在 GPT 编辑器 → Actions 里粘贴它。之后 GPT 会调这个 Action 出分，不再靠 Code Interpreter 估 —— **每次都一致**。

## <a name="q23"></a>Q23：把 Gemini 加到主页

**Claude：**

把 Gemini 卡片点亮。三张卡现在都点亮了：ChatGPT、Claude、Gemini。Gemini 卡链接到 Opal app，标签改成了「Google Opal App」。

但有个**必须正视的一致性问题**：

Opal 是把提示词、Gemini 模型和谷歌自带工具串成工作流的东西，**它对外部 API 的连接很有限**。换句话说 —— 这个 Gemini 版跑的是 Gemini 的"推理估分"，**并没有跑你那套 `score.py` 引擎**。

后果很实在：它给的分是模型估的，**不可复现**，而且大概率跟 ChatGPT / Claude / 网页版的 97 对不上。而你主页"为什么可信"里白纸黑字写着"可复现：同样的输入，永远同样的输出" —— **三个平台分数打架，这条信任承诺当场就破了**。

**Opal 限制内的最佳解法**：把参考表和打分公式直接压进 Analyze 那一步的提示词里，让 Gemini 对着同一套数字、同一个公式、一步步算。这样能贴得很近 —— 但老实说，模型做算术还是可能有零点几的漂移，做不到和引擎 100% 等同。所以要么走这条（尽量逼近），要么就把 Gemini 这版定位成"快速/轻量版"并如实标注，**别宣称和引擎版完全一致**。

**两个落地提醒**：

- 网址需要 Share App 拿到真正的分享链接，编辑器地址别人打不开。
- Opal 早期是**美国限定的公开 beta**，访客要用自己的 Google 账号登录才能用。挂上主页前先确认你的目标用户（尤其国内）真能打开。

## 核心设计原则汇总

这场对话沉淀下来 7 条贯穿始终的设计纪律：

1. **"客观"不是目标，"透明 + 可复现 + 独立"才是。** 客观本不存在，强行声称是法律陷阱。
2. **打分必须是确定性规则引擎，不是 LLM 估分。** Claude 负责对话和解释，引擎负责算分 —— 同输入同输出，可审计。
3. **事实与权重分离。** 参考表挂可引用来源；权重是编辑选择，公开声明，自己署名。
4. **诚实标注未覆盖。** 不在参考表里的成分必须显式标"未计入评分、覆盖几分之几"，禁止静默忽略。
5. **范围按格子上匝道。** 成分类别 × 市场 × 垂类 三轴别同时推。挑最强一格站稳，再一次只延一根轴。
6. **模型是租来的层，资产在它下面。** 私有数据、独立信任品牌、分发网络、经校准的方法学 —— 这才是护城河。
7. **合规是动作前置校验，不是末端过滤器。** 按 法域 × 语言 切规则，宁可保守，绝不靠"翻译正则"。
